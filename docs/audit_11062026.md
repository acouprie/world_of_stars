# Audit de sécurité — World of Stars

**Date :** 2026-06-11
**Périmètre :** application Rails 8.1 (auth, injections, durcissement web, abus de logique de jeu)
**Méthode :** revue manuelle + 3 sous-agents d'exploration + Brakeman 8.0.4 + bundler-audit
**Branche auditée :** `feature/implement_units`

---

## Synthèse

L'application est **globalement saine**. Brakeman ne remonte **aucun avertissement**, bundler-audit **aucune dépendance vulnérable**, et les fondamentaux Rails sont respectés : sessions serveur signées, `params.permit` strict partout, requêtes ActiveRecord paramétrées, IDOR correctement bloqué (tout passe par `Current.user.planets.find`), CSRF actif, mots de passe bcrypt avec politique de complexité.

⚠️ **Important — deux « findings critiques » remontés par l'analyse automatique sont des faux positifs** que j'ai vérifiés et écartés (détail en annexe). Les vrais points à traiter sont des durcissements de priorité moyenne à basse, pas des failles exploitables aujourd'hui.

| # | Sujet | Sévérité réelle | Action |

| 1 | CSP désactivée (initializer commenté) | Moyenne | Activer une CSP |

## Vrais points à traiter

### 1. Content Security Policy désactivée — Moyenne
`config/initializers/content_security_policy.rb` est entièrement commenté (défaut Rails). Aucun en-tête CSP n'est émis. En cas de XSS DOM ou de compromission d'une lib tierce (le front utilise React/Pixi), rien ne limite l'exécution de script. C'est une défense en profondeur manquante, pas une faille active.

**Reco :** activer une CSP, au minimum `default-src :self`, `object-src :none`, `script-src`/`style-src` en `:self` + nonce. À tester en mode report-only d'abord pour ne pas casser Vite/Pixi.

### 2. Throttle `rack-attack` de l'API advisor inopérant — Moyenne (latent)
`config/initializers/rack_attack.rb` :
```ruby
throttle("advisor/user", limit: 1, period: 1.minute) do |req|
  req.env["warden"]&.user&.id if req.path.start_with?("/api/advisor")
end
```
L'application **n'utilise pas Warden/Devise** mais une authentification maison (`Current.session`). `req.env["warden"]` est donc toujours `nil`, la clé de throttle vaut `nil`, et **la limitation ne s'applique jamais**. L'endpoint `/api/advisor` n'existe pas encore (ni contrôleur ni route), donc l'impact est nul aujourd'hui, mais le garde-fou sera silencieusement inactif le jour où il sera branché — alors que l'archi prévoit explicitement 1 appel/minute (coût LLM probable).

**Reco :** baser la clé sur l'auth réelle, p. ex. `Current.session`/cookie signé, ou throttler par IP. À corriger **avant** d'exposer l'endpoint.

### 3. `Api::PlanetsController#index` renvoie tous les joueurs — Basse
`app/controllers/api/planets_controller.rb` renvoie pour **toutes** les planètes : coordonnées, biome, `user_id` et `user_name`. Pour une carte galactique multijoueur, exposer les positions et le nom du propriétaire est normal et probablement voulu. À confirmer : si l'intention est que le brouillard de guerre / l'espionnage masque certaines infos, alors cet endpoint les court-circuite. Sinon, RAS.

**Reco :** confirmer l'intention de design. Si masquage attendu, scoper les champs sensibles.

### 4. Cookie `locale` sans drapeaux — Basse
`app/controllers/locales_controller.rb` : `cookies.permanent[:locale] = locale` est posé sans `httponly`/`same_site`, contrairement au cookie de session (`httponly: true, same_site: :lax`). Impact faible (la valeur est validée contre `I18n.available_locales` avant écriture et relue défensivement), mais incohérent.

**Reco :** `cookies.permanent[:locale] = { value: locale, httponly: true, same_site: :lax }`.

### 5. Énumération de comptes par timing sur le reset — Basse
`passwords_controller.rb#create` renvoie le même message dans les deux cas (bien), mais ne fait le travail d'envoi que si l'email existe, d'où une légère différence de temps de réponse. Atténué par le rate-limit (10/3 min). Théorique.

**Reco (optionnel) :** uniformiser le chemin d'exécution.

### 6. Resolvers combat / espionnage / exploration non câblés — Préventif
`Combats::Resolver`, `Espionnages::Resolver`, `Explorations::Resolver` existent mais ne sont reliés à **aucun contrôleur**. Le moteur de combat est pur et bien isolé (bon design). **Au moment du câblage**, vérifier impérativement côté serveur :
- que l'attaquant possède bien les unités engagées (`Current.user.planets`…) ;
- l'impossibilité de rejouer une attaque / de se cibler soi-même pour farmer ;
- que le rapport d'espionnage ne fuit pas d'infos au-delà du niveau d'espionnage atteint.

C'est le principal risque **futur** de la base de code : tant que ces features ne sont pas exposées, rien à corriger, mais elles devront recevoir le même soin d'ownership que les contrôleurs existants.

## Ce qui est bien fait (à conserver)

- **IDOR** : chaque accès objet passe par `Current.user.planets.find(params[:id])` — vérifié sur planets/buildings/construction_queues/training_queues.
- **Mass assignment** : `params.permit` strict partout, aucun `permit!`, aucun champ privilège (role/admin/credits) exposé.
- **Injections SQL** : aucune interpolation de chaîne dans `where/order`, aucun `find_by_sql`/`execute`. Brakeman = 0.
- **Double-dépense / TOCTOU** : déductions de ressources encadrées par `planet.with_lock` + transaction ; quantités négatives bloquées (`@quantity > 0`).
- **Jobs Sidekiq idempotents** : `CompleteConstructionJob` / `CompleteTrainingJob` re-vérifient `pending?` après `reload`.
- **Coûts et temps de complétion calculés côté serveur** (registres `Buildings`/`Units`), jamais reçus du client.
- **Reset mot de passe** : tokens signés Rails 8.1 avec expiration, invalidation de **toutes** les sessions (`@user.sessions.destroy_all`).
- **Rate-limiting login** (5/min/IP) effectif + Allow2Ban.
- **Secrets** : `.env` est bien **gitignoré**, secrets via `credentials.yml.enc` ; `config/master.key` **non** versionné.

## Annexe — Faux positifs écartés (avec vérification)

L'analyse automatique a remonté deux « CRITICAL » qui **n'en sont pas**. Documenté ici pour éviter qu'ils ne ressortent.

### ❌ « Secrets commités dans `.env` » → FAUX
Un agent a lu un fichier `.env` local contenant des identifiants et a conclu à une fuite critique. Vérification :
```
$ git check-ignore .env   → .env (ignoré)
$ git ls-files | grep '^\.env'  → .env.example uniquement
```
Le `.env` réel est **gitignoré** et n'a jamais été versionné (aucun historique). Seul `.env.example` est suivi, avec des valeurs d'exemple (`POSTGRES_PASSWORD=password`, `RAILS_MASTER_KEY=` vide) — ce qui est l'usage correct. **Aucun secret n'est exposé dans le dépôt.**

### ❌ « Remboursement total à l'annulation = duplication de ressources » → FAUX
`constructions/cancel_service.rb` rembourse 100 % du coût à l'annulation. L'agent y a vu une duplication critique. En réalité c'est **net nul** : le joueur a dépensé N métal, il récupère N métal. Aucune ressource créée. Le seul « gain » est la production normale écoulée pendant la construction, qui a lieu de toute façon. C'est un **choix de design** (annuler ne coûte que du temps), pas une faille. À transformer en remboursement partiel **seulement si** le game design l'exige — ce n'est pas un problème de sécurité.
